# Building & Architecture Pipeline

FzBoard utilizes a modern, hybrid architecture merging a dynamic frontend widget capability (Svelte/Vite) with a hardened backend orchestrator (Django/Gunicorn/Kubernetes). To ship optimized payloads into production without dragging development tools directly into the Kubernetes network, the building pipeline executes in rigorous, highly isolated layers.

## The Multi-Stage Dockerfile Strategy

The primary pipeline orchestrator is our tightly tracked `Dockerfile`. It embraces a **Multi-Stage Build Paradigm** so our final live production images are significantly smaller, radically faster, and intrinsically more secure.

### Stage 1: The Builder (Frontend Assets)
The builder leverages a lightweight bun/vite layer (`oven/bun`).
1. **Dependency Injection:** It pulls down all `node_modules` and JS libraries needed to compile `Svelte`.
2. **Context Passing:** The `frontend/` directory (holding Vue/Svelte files) along with `dashboard/` templates are fully exposed to the internal container logic.
3. **Tailwind compilation:** `bun run build` scans dynamically. The internal static analyzer of Tailwind CSS deeply crawls over the `dashboard/templates/**/*.html` to strictly guarantee only the styles natively deployed to the `login.html` form remain in the final `CSS` payload window.
4. **Conclusion:** It exports the fully minified static payloads into a `/app/static/dist/assets` directory, completely ignoring `.env` logs or heavy compilers.

### Stage 2: The Production Backend
The secondary container is spun up natively over `python:3.12-slim` focusing purely on orchestrating Python functionality.
1. **Library Extraction:** All python modules like `Whitenoise` or `django-vite` are instantiated and isolated.
2. **Sanitation:** Before executing `collectstatic`, any garbage frontend files that may have been polluted from the host's directory during local development are scrubbed off the disk (`rm -rf /app/static/dist /app/db.sqlite3`).
3. **Payload Merger:** The fully hardened static `dist/assets` block generated by Stage 1 is transferred via `COPY --from=builder` into the Django application image.
4. **Runtime Bootstrap:** The container starts through `docker-entrypoint.sh`, which creates the SQLite and media parent directories, runs `python manage.py migrate --noinput`, and only then hands over control to Gunicorn. This removes the old first-boot failure on blank persistent volumes.
5. **Security Hardening:** A non-root user `appuser (1000)` enforces restricted file handling execution to ensure no internal file system corruption executes via a rogue shell.

## Static Resolution

Once inside production, Vite uses dynamic hashing on all CSS/JS modules (such as `main-XYZ.css`). Therefore, pure HTML templates must rely on dynamic manifests.
1. When Gunicorn (`WSGI`) initially executes, `django-vite` pulls the final mapping matrix from `.vite/manifest.json` completely into running RAM layout. 
2. Subsequent hits on templates bypass compilation seamlessly.
3. `WhiteNoise`, seamlessly slotted directly into the center of the Django core middleware stack, immediately catches the URL requests for `/static/*`. It streams them back out highly aggressively utilizing deep HTTP caching hooks and `Brotli` compressions, eliminating the necessity to layer `Nginx` entirely in front of the process for small-to-medium network architectures.

## Helm Infrastructure Pipeline
FzBoard relies natively on `Helm` templates mapping over `Kubernetes` deployments.
1. The chart mounts a `PersistentVolumeClaim` at `/app/data`, keeping the SQLite database at `/app/data/db/db.sqlite3` and user uploads under `/app/data/media`.
2. An init container prepares the mounted directories and correct ownership before the Django container starts, which is important because Minikube-backed volumes may come up empty or root-owned.
3. The application container then performs its migration bootstrap during startup, so a brand-new PVC becomes immediately usable without a manual `kubectl exec ... migrate` recovery step.
4. The deployed image is exposed through a `ClusterIP` service and optional ingress routing.
